# localizator

CLI-приложение и rendezvous-сервер для прямого подключения в локальную
сеть в обход NAT — P2P-туннель с UDP hole punching и автоматическим
fallback на relay через сервер, если punching не удался.

Два режима:

- **port-forward** (`connect`/`expose`) — пробросить один порт с одной
  машины на другую, как `ngrok`/`frp`.
- **mesh VPN** (`join`) — полноценная сетевая связность между всеми
  участниками сети через реальный TUN-интерфейс и WireGuard, как
  `tailscale`.

Весь трафик данных шифруется end-to-end по ключам участников независимо
от того, прошло соединение напрямую (P2P) или через relay-сервер —
сервер физически не может прочитать содержимое.

## Установка

Собрать из исходников (нужен Go 1.24+):

```sh
git clone <адрес репозитория>
cd localizator
make build          # соберёт ./bin/app под текущую платформу
```

Релизные бинарники под все платформы: `make release` (см. `make help`).

## Быстрый старт

### 1. Поднять rendezvous-сервер

На машине с публичным IP:

```sh
app server --listen :4443 --stun-listen :4444
```

Сервер — это чистый rendezvous/signaling: он не проксирует данные
пользователей (кроме fallback-relay, когда punching не сработал), только
знакомит клиентов между собой. Состояние (зарегистрированные пиры,
mesh-сети и их участники) хранится в SQLite и переживает рестарт сервера
(по умолчанию `~/.config/localizator/state.db`, переопределяется флагом
`--db`). Сервер пишет структурные логи в stderr (регистрации, join
сетей, ошибки на каждый RPC) — по умолчанию уровня info, флаг `--verbose`
добавляет debug.

### 2. Узнать свой peer-id

Без обращения к сети (используется локально сохранённая идентичность):

```sh
app whoami
```

Идентичность (X25519-ключ) сохраняется в `~/.config/localizator/identity.key`
и переиспользуется между запусками — без этого клиенты не смогли бы
заранее узнать id друг друга до первого подключения.

### 3. Пробросить порт (`connect` / `expose`)

На машине, где работает сервис (например, SSH на 22 порту):

```sh
app expose --server SERVER:4443 --stun-server SERVER:4444 \
  --to <peer-id того, кто будет подключаться> --port 22
```

На машине, которая хочет подключиться:

```sh
app connect --server SERVER:4443 --stun-server SERVER:4444 \
  --to <peer-id сервиса выше> --local-port 2222
```

После этого `localhost:2222` пробрасывается на `22` порт удалённой
машины.

### 4. Mesh VPN (`join`)

Требует root/`CAP_NET_ADMIN` — команда создаёт реальный TUN-интерфейс
(`localizator0`) и поднимает WireGuard.

Первый участник создаёт сеть и получает инвайт-токен:

```sh
sudo app join --server SERVER:4443 --stun-server SERVER:4444 \
  --network home
```

Вывод содержит `инвайт-токен` — передайте его остальным участникам:

```sh
sudo app join --server SERVER:4443 --stun-server SERVER:4444 \
  --network home --invite <токен>
```

Каждому участнику назначается адрес из `100.64.0.0/10`; после запуска все
участники сети видят друг друга напрямую через TUN-интерфейс, как в
обычной VPN.

Чтобы проверить сетевую координацию без поднятия TUN (не требует root):
`app join-network` с теми же флагами.

## Конфиг-файл

Чтобы не повторять `--server`/`--stun-server`/`--identity` в каждой
команде, можно один раз задать их в `~/.config/localizator/config.json`
(на macOS/Windows — в соответствующей конфиг-директории пользователя):

```json
{
  "server": "SERVER:4443",
  "stun_server": "SERVER:4444"
}
```

Флаги командной строки всегда имеют приоритет над конфиг-файлом; сам файл
опционален — его отсутствие ничего не меняет в поведении по умолчанию.

## Команды

| Команда | Что делает |
|---|---|
| `app server` | Rendezvous-сервер (control-plane + STUN + relay fallback) |
| `app whoami` | Свой peer-id, без сети |
| `app register` | Диагностика: зарегистрироваться на сервере, показать наблюдаемый адрес |
| `app connect` | Пробросить локальный порт на сервис пира (`expose`) |
| `app expose` | Открыть локальный сервис пиру (`connect`) |
| `app join` | Присоединиться к mesh-сети (реальный TUN + WireGuard) |
| `app join-network` | Диагностика: та же координация сети, без TUN |
| `app version` | Версия |

У каждой команды — `--help` со списком флагов.

## Безопасность

- **Идентичность** — X25519-keypair, персистентный (`~/.config/localizator/identity.key`).
- **Control-канал** — QUIC/TLS с trust-on-first-use пиннингом сертификата
  сервера (`~/.config/localizator/known_servers.json`): первое подключение
  к серверу ничем не защищено (как SSH host key), но подмена сервера
  *после* того, как он уже был доверен, обнаруживается.
- **Data-plane** — end-to-end шифрование (X25519 ECDH → HKDF → AES-256-GCM)
  поверх любого транспорта (P2P или relay), независимо от WireGuard,
  который в mesh-режиме и так шифрует сам.
- **Mesh-сети** защищены инвайт-токеном: чтобы присоединиться к уже
  существующей сети, нужно знать токен, который выдаётся при её создании.

Подробности архитектуры, известные ограничения и решения по каждой фазе
разработки — в `CLAUDE.md`.

## Разработка

```sh
make test     # go test ./... -race
make vet      # go vet + gofmt -l
make build    # собрать ./bin/app под текущую платформу
```

Изменение control-протокола требует перегенерации protobuf-кода:

```sh
make proto    # нужны protoc и protoc-gen-go в PATH
```
