# spur

CLI-приложение + сервер для прямого подключения в локальную сеть в обход NAT.
Модель: P2P-туннель с UDP hole punching, с fallback на relay через сервер, если
punching не удался (symmetric NAT). Поддерживает два режима работы: проброс
одного порта (port-forward) и полноценный mesh VPN (WireGuard-туннель).

Этот файл — единственный источник правды об архитектуре и правилах работы
над проектом. Перечитывай его в начале каждой сессии.

## Архитектура

Три логических компонента, один Go-модуль, **два бинарника** из одного и
того же дерева кода — `spur` (клиент: `connect`, `expose`, `join`,
`whoami`, ...) и `spur-server` (сервер, флаги на корневой команде, без
подкоманды). Раньше это был один бинарник с подкомандой `spur server`; см.
«Разделение клиента и сервера» ниже за тем, почему и как это разделилось
без нарушения слоёв Clean Architecture:

1. **Rendezvous/Signaling-сервер** (публичный IP). Не проксирует трафик по
   умолчанию — знакомит клиентов и помогает пробить NAT. Держит relay как
   fallback.
2. **CLI-клиент** — работает в локальной сети за NAT, регистрируется на
   сервере, пробивает P2P-канал к другому клиенту либо получает канал через
   relay.
3. **Data plane** — после установления канала трафик идёт либо как
   port-forward (один сервис), либо как полноценный TUN-интерфейс (mesh) —
   в зависимости от режима запуска.

### Установление соединения (ICE-подобная схема)

```
Клиент A (за NAT)                 Сервер (публичный IP)              Клиент B (за NAT)
      |--- control-канал (QUIC) ------>|<--- control-канал (QUIC) ------|
      |        register + STUN         |         register + STUN        |
      |<---- public ip:port (A) -------|----- public ip:port (B) ------>|
      |                                 |                                |
      |<====== обмен candidate-листами через сервер (signaling) =======>|
      |                                 |                                |
      |========= UDP hole punching (одновременные пакеты) ==============|
      |                                 |                                |
      punching не удался (symmetric NAT) → relay через сервер (TURN-подобно)
```

- STUN-часть сервер отдаёт бесплатно: клиент делает binding request, узнаёт
  свой public `ip:port`, как его видит NAT.
- Кандидаты: host (локальный IP), server-reflexive (через STUN), relay
  (через сервер) — упрощённая версия ICE, без полноценного SDP.
- Hole punching: оба клиента одновременно шлют UDP на public-адрес друг
  друга.
- Relay fallback — сервер гоняет байты между двумя control-соединениями.
  Единственный случай, когда сервер физически видит трафик — поэтому data
  plane должен быть end-to-end зашифрован независимо от транспорта
  (relay не должен мочь прочитать полезную нагрузку).

### Data plane — два режима

**Port-forward** (`spur connect --to peer-id --local-port 2222 --remote-port 22`)
Поверх установленного канала — один мультиплексированный поток
(QUIC-стримы), клиент слушает локальный TCP-порт и проксирует в один
сервис на удалённой стороне.

**Mesh VPN** (`spur join --network home`)
Клиент поднимает TUN-интерфейс, каждому узлу — адрес из приватной подсети
(аналог Tailscale, `100.64.x.x`). Внутри — обычный WireGuard
(`wireguard-go`), P2P-канал из шага выше используется как транспорт для
UDP WireGuard-пакетов, endpoint динамически обновляется после punching.
Сервер в этом режиме — control plane в духе Tailscale coordination server:
раздаёт список пиров, публичные ключи, актуальные endpoint'ы.

Оба режима переиспользуют один и тот же rendezvous/punching слой —
различается только то, что происходит с байтами после установления канала.

### Технологический стек

| Компонент | Библиотека |
|---|---|
| CLI | `spf13/cobra`. Конфиг-файл — не `viper` (см. секцию «Фаза 8» ниже, почему) |
| Control-протокол клиент↔сервер | `protobuf`-сообщения в length-prefixed фреймах поверх QUIC-стримов (`quic-go`). Полноценный `grpc` не используется: его транспорт завязан на HTTP/2-фрейминг, который не работает поверх сырого QUIC без HTTP/3-прослойки — вместо этого свой минимальный RPC поверх `quic-go` стримов, кодогенерация сообщений через `protoc` + `protoc-gen-go` |
| STUN | `pion/stun` |
| Мультиплексирование (port-forward) | нативные QUIC-стримы |
| Mesh-туннель | `wireguard-go` (`golang.zx2c4.com/wireguard`) |
| Шифрование control-канала | TLS 1.3 (встроен в QUIC) + токены аутентификации |
| Хранение состояния сервера | SQLite (`modernc.org/sqlite`, без cgo) — `adapter/repository/sqlite`, персистентно между рестартами (см. «Фаза 8+: SQLite» ниже). Ephemeral in-flight состояние (candidate exchange, relay-сплайсинг) остаётся в `adapter/repository/memory` — персистить нечего по своей природе |
| Логи | `rs/zerolog` — `controlserver.Server.Logger` (см. «Фаза 8+: логирование» ниже). CLI-команды (`connect`/`expose`/`join`/...) по-прежнему используют `cmd.Printf` для пользовательского вывода — это статус для человека за терминалом, а не операционные логи долгоживущего процесса, подключать туда zerolog не было смысла |
| Тесты | стандартный `testing` + `testify/require` для ассертов |

### Требования окружения для сборки

- Go (проверено на 1.26.4) должен быть в `PATH`.
- Для регенерации `*.pb.go` из `.proto` нужны в `PATH`: `protoc` (системный
  пакет) и `protoc-gen-go` (`go install
  google.golang.org/protobuf/cmd/protoc-gen-go@latest`, ставится в
  `$(go env GOPATH)/bin`). Сгенерированные `.pb.go` коммитятся в репозиторий,
  чтобы обычная сборка (`go build`) не требовала `protoc` — он нужен только
  при изменении `.proto`-файлов.
- TLS сервера в Фазе 2 — самоподписанный сертификат, генерируется на лету,
  клиент временно отключает проверку цепочки (`InsecureSkipVerify`). Это
  осознанное временное упрощение — верификация (pinning по публичному ключу
  сервера) закрывается в Фазе 7, не раньше. Не забыть об этом при доведении
  проекта до чего-то, что можно выставлять наружу.

### Безопасность

- End-to-end шифрование данных не зависит от того, идёт трафик P2P или
  через relay.
- Аутентификация узлов: keypair (аналог WireGuard) генерируется при первом
  запуске клиента, сервер знает только публичные ключи. Вход в сеть — по
  инвайт-токену.
- Сервер как rendezvous не требует доверия для конфиденциальности, только
  для доступности.

## Clean Architecture — обязательное правило структуры кода

Весь код обязан следовать Clean Architecture (слои + правило зависимостей:
зависимости смотрят только внутрь, внутренние слои не знают о внешних и не
импортируют их). Границы между слоями — через интерфейсы (ports), которые
определяет внутренний слой, а реализует внешний (dependency inversion).

```
/cmd/spur                        — клиентский composition root (whoami/connect/expose/join/...)
/cmd/spur-server                     — серверный composition root (control-plane + STUN + relay fallback)
                                   Оба — entrypoints, DI-сборка (frameworks & drivers, самый внешний
                                   слой); два разных main-пакета, ни один не импортирует другой.

/internal/domain                — Entities. Чистые структуры без внешних зависимостей:
                                   Peer, Network, Candidate, Session, Tunnel.
                                   Никаких импортов из adapter/infra/framework-библиотек.

/internal/usecase                — Use Cases / Application Business Rules.
    /usecase/port                — интерфейсы, которые нужны юзкейсам
                                   (Repository, Signaler, Puncher, Relay, TunnelTransport).
                                   Юзкейсы зависят только от domain и от своих же port-интерфейсов,
                                   никогда — от конкретных реализаций в adapter/infra.
    RegisterPeer, DiscoverPeer, EstablishTunnel, PunchNAT, RelayFallback,
    JoinNetwork, ForwardPort — по одному use case на файл/пакет.

/internal/adapter                — Interface Adapters. Реализуют порты из usecase,
                                   переводят данные между внешним миром и domain-моделями.
    /adapter/controlserver         — control-plane обработчики на сервере (QUIC), диспетчеризация по Method
    /adapter/controlclient         — control-plane клиент (QUIC), реализует usecase/port.Signaler
                                     и usecase/port.Relay (relay-канал — это просто ещё один стрим
                                     на этом же соединении)
    /adapter/controlproto          — protobuf-сообщения, фрейминг и Method-теги control-протокола
    /adapter/stunserver            — RFC5389 STUN Binding responder (отдельный UDP-порт от control-plane)
    /adapter/repository/memory    — in-memory реализации port.CandidateStore и port.RelayBroker
                                     (по природе недолговечное, in-flight состояние — персистить
                                     нечего, см. секцию «Фаза 8+: SQLite» ниже); также используется
                                     как лёгкий тестовый дублёр port.PeerRepository/NetworkRepository
                                     в юнит-тестах usecase/adapter слоёв, где реальный SQLite-файл
                                     избыточен
    /adapter/cli                  — cobra-команды, транслируют CLI-флаги в вызовы usecase
    /adapter/repository/sqlite    — implements port.PeerRepository и port.NetworkRepository,
                                     персистентное состояние сервера (see «Фаза 8+: SQLite» ниже)
    /adapter/nat                  — STUN-клиент (та же UDP-сокета, что и punching), host-кандидаты,
                                     implements usecase/port.Puncher
    /adapter/localnet              — TCP-реализации port.LocalListener/port.LocalDialer
                                     (локальная сторона port-forward: `connect` слушает, `expose` дозванивается)
    /adapter/tunnel                — implements usecase/port.TunnelTransport: QUIC поверх пробитого
                                     UDP-сокета для P2P, yamux поверх relay-стрима для relay;
                                     wireguard-go для mesh добавится в Фазе 6

/internal/infra                  — низкоуровневая обвязка: конфиг, логгер, TLS/QUIC setup.
```

Правила, которые нельзя нарушать:
- `domain` не импортирует ничего из `usecase`, `adapter`, `infra`, `cmd` и
  сторонних библиотек (кроме stdlib).
- `usecase` не импортирует `adapter`/`infra`/`cmd`, не импортирует
  конкретные библиотеки (`quic-go`, `wireguard-go`, `pion/stun` и т.п.)
  напрямую — только через свои `port`-интерфейсы.
- `adapter` может импортировать `usecase` (реализует его порты) и `domain`,
  но не другие пакеты `adapter` напрямую, если это не оправдано (например
  `adapter/cli` вызывает usecase, а не другой adapter).
- Вся сборка графа зависимостей (какая реализация порта подставляется) —
  только в composition root (`cmd/spur` для клиента, `cmd/spur-server` для
  сервера).
- Каждый новый usecase — сначала пишется интерфейс порта и сам usecase с
  тестами на моках, потом уже пишется adapter-реализация.

### Тестирование сетевых адаптеров (UDP/QUIC)

Функции `Serve` в `controlserver` и `stunserver` принимают уже забинженный
`net.PacketConn`/`*net.UDPConn`, а не строку адреса. Это не только удобнее
для передачи уже созданных сокетов, но и убирает гонку в тестах: если
`Serve` сам резолвит и биндит адрес внутри себя, тест не может узнать
реальный (эфемерный, `:0`) порт до того, как горутина с сервером
запустится, и вынужден либо угадывать порт заранее (коллизии между
прогонами, дефолтный `go test -count=N` их быстро ловит), либо спать
после старта (медленно и всё равно не гарантирует готовность). Биндить
сокет синхронно в тесте до запуска горутины, затем передавать готовый
`conn` в `Serve` — единственный вариант без гонки. Держаться этого паттерна
для любых будущих сетевых адаптеров (relay, tunnel).

### Время жизни relay-стрима (важно для Фаз 5/6)

`port.Relay.OpenChannel` возвращает `io.ReadWriteCloser`, но у реализации
в `controlclient` это конкретно `*quic.Stream`, открытый на control-канале
клиента. Стрим живёт ровно столько, сколько жива QUIC-connection, на
которой он открыт — если закрыть `*controlclient.Client` (control-канал),
все его стримы, включая relay, обрываются вместе с ним. При построении
data-plane поверх relay (Фаза 5/6) нельзя закрывать control-соединение,
пока relay-канал используется; нужно либо держать его открытым на весь
срок жизни тоннеля, либо явно связать время жизни обоих (как это сделано
в тесте `establish_session_e2e_test.go`, см. `relayStreamWithConn`).

### Bootstrap идентичности: `spur whoami` и `--identity`

`domain.SessionIDFor(a, b)` детерминирован, но требует, чтобы **обе**
стороны знали ID друг друга ДО начала rendezvous — сервер не сводит
`connect` и `expose` за них. Пока ключи были чисто эфемерными (новая
случайная пара на каждый запуск), это делало флоу невозможным на
практике: пользователь физически не мог узнать ID ещё не запущенного
процесса. Решение — `infra.LoadOrCreateIdentity` персистит identity на
диск (`~/.config/spur/identity.key` по умолчанию, флаг
`--identity` у `connect`/`expose`/`whoami`/`join`/`join-network`), а
`spur whoami` печатает свой ID без обращения к сети. Реальный флоу: обе
стороны по отдельности запускают `spur whoami`, обмениваются ID (пока —
вручную, out-of-band), потом запускают `connect --to <id-пира>` /
`expose --to <id-пира>`. Discovery/invite-token решение, снимающее ручной
обмен ID, по-прежнему запланировано отдельно — см. дорожную карту.

С Фазы 7 `identity.key` хранит настоящий приватный X25519-ключ
(`crypto/ecdh`), а не случайные байты без криптографического смысла:
`infra.Identity{PrivateKey, PublicKey}`, публичный ключ всегда выводится
из приватного, а не хранится отдельно. Это было обязательным условием для
mesh-режима: WireGuard делает свой Noise-handshake и ему нужна настоящая
согласованная пара ключей, а не два независимых случайных блоба.

### End-to-end шифрование data plane (`adapter/e2e`)

`domain.CandidateSet` (кандидаты + публичный ключ) публикуется вместе при
обмене кандидатами (`PublishCandidatesRequest`/`AwaitCandidatesResponse`
несут `public_key`), так что к моменту установления сессии обе стороны уже
знают настоящий публичный ключ друг друга — не только ID.

`adapter/e2e.WrapConn(tunnelConn, myPriv, peerPub, isDialer)` оборачивает
`port.TunnelConn` вне зависимости от того, P2P он или relay: ECDH
(`myPriv.ECDH(peerPub)`) → `crypto/hkdf` с разными info-метками на
направление (`to-dialer`/`to-listener`, гарантирует, что send-ключ
одной стороны равен recv-ключу другой без доп. согласования) → AES-256-GCM
с монотонным nonce-счётчиком на направление, кадры length-prefixed.
Подключено в `cmd/spur/tunnel.go`'s `rendezvous()`, то есть автоматически
работает и для port-forward (`connect`/`expose`), и для mesh (`join`,
который переиспользует ту же `rendezvous()`).

Зачем это нужно ОБОИМ путям, не только relay: relay ожидаемо инсекьюрен
(`memory.RelayBroker` сплайсит стримы в открытую — сервер физически видит
байты), но P2P-путь тоже не полностью доверенный, пока клиент верифицирует
TLS-сертификат сервера через `InsecureSkipVerify` (см. следующий абзац) —
`adapter/e2e` закрывает оба случая одним механизмом, независимо от
транспорта под ним.

### TOFU-пиннинг сертификата control-plane сервера

`infra.InsecureClientTLSConfig` (слепое доверие) для control-соединения
заменён на `infra.TOFUClientTLSConfig`: сервер получает ПЕРСИСТЕНТНЫЙ
сертификат (`infra.LoadOrCreateServerTLSConfig`, хранится рядом с
identity, переживает рестарт — иначе TOFU-пиннинг ловил бы каждый
рестарт сервера как «сервер подменился») вместо эфемерного
(`SelfSignedServerTLSConfig` остался для data-plane P2P-хендшейка, где
это не нужно — `adapter/e2e` уже аутентифицирует по-настоящему на этом
уровне). Клиент запоминает fingerprint сертификата сервера при первом
подключении (`~/.config/spur/known_servers.json`, ключ —
`serverAddr`) и требует точного совпадения на последующих — при
расхождении явная ошибка вместо тихого MITM. Классическое ограничение
TOFU: самое первое подключение к серверу ничем не защищено (как SSH
host key при первом коннекте) — но подмена сервера ПОСЛЕ того, как он
уже был доверен, теперь ловится.

При живой проверке всплыл ещё один race: `connect` и `expose` —
отдельные процессы, оба пиннящие один и тот же сервер почти
одновременно — конкурентно писали в один файл `known_servers.json`.
`os.WriteFile` truncate'ит файл перед записью, и читатель, попавший в
это окно, получал `unexpected end of JSON input`. Фикс — атомарная
запись (temp-файл в той же директории + `os.Rename`, что атомарно на
POSIX-файловых системах) в `saveTrustStore`, с регрессионным тестом,
гоняющим конкурентных писателей и читателей одновременно.

### Инвайт-токены для `JoinNetwork`

`usecase.JoinNetwork` теперь различает создание сети и вход в
существующую: если сети с таким именем ещё нет — создающий её получает
случайный `InviteToken` (16 байт, hex) в ответе и не обязан ничего
предъявлять; если сеть уже существует и вызывающий ещё не её участник —
обязателен корректный `invite_token` в запросе, иначе
`domain.ErrInvalidInviteToken`. Уже известным участникам токен для
повторного join не нужен (idempotent-повтор не требует повторного
доказательства). Сервер возвращает токен в `JoinNetworkResponse` каждому,
кто успешно вошёл (не только создателю) — им можно поделиться с
следующим человеком. CLI: `--invite` у `spur join` и `spur join-network`;
`join-network` печатает токен в выводе при успехе.

В отличие от остальных RPC (которые на ошибке молча рвут стрим —
`handleRegister`, `handlePublishCandidates` и т.д.), `JoinNetworkResponse`
получил явное поле `error`: неверный токен — частый случай опечатки
пользователя, и тут действительно хотелось вменяемое сообщение, а не
голый обрыв соединения. Это единственный RPC с таким полем; остальные
пока намеренно оставлены как есть (не расширять паттерн без необходимости).

Также: `cobra.Command.Print/Println/Printf` пишут в **stderr** (не
stdout) — это поведение библиотеки, не опечатка. Для вывода, который
должен быть перехватываемым (`id=$(spur whoami)`), используй
`fmt.Fprintln(cmd.OutOrStdout(), ...)` явно, как в `whoami.go` и
`version`-команде. Статусные сообщения (`server`, `connect`, `expose`) —
можно оставлять на `cmd.Printf`, там stderr уместен.

### Не забывай сбрасывать `SetReadDeadline` перед передачей сокета дальше

Реальный баг, найденный при ручной проверке Фазы 5 (два отдельных
CLI-процесса «connect»/«expose», а не горутины в одном тесте): `nat.UDPPuncher.recvLoop`
крутит `SetReadDeadline(now + 200ms)` в цикле опроса и — до фикса — ни разу
не сбрасывал дедлайн на выходе. `adapter/tunnel.Transport` переиспользует
тот же `*net.UDPConn` для QUIC сразу после успешного punching (это
единственный способ не потерять пробитую NAT-мэппинг). В результате
дедлайн оставался в прошлом, и КАЖДОЕ чтение quic-go на этом сокете сразу
падало с `i/o timeout` вместо блокировки — внешне выглядело как: CPU
процесса `connect`/`expose` улетал на 100%+ и не опускался, `ss -uapn`
показывал огромный неубывающий Recv-Q на обеих сторон, при этом
`EstablishSession` рапортовал `established_p2p` (сам QUIC-handshake
успевал проскочить), но `AcceptStream`/дальнейшие данные зависали
навсегда. Автотесты (`go test`, включая `-race`) этого не ловили — там всё
успевает произойти за миллисекунды, до того как протухший дедлайн
проявит себя. Отдельный автономный skeleton (голый quic-go, два процесса,
без punching) подтвердил, что сам quic-go в этом окружении работает
нормально — баг был именно в не сброшенном дедлайне.

Фикс — в `nat.UDPPuncher.Punch`: дождаться (`sync.WaitGroup`), пока
`sendLoop`/`recvLoop` реально завершатся, и только потом
`SetReadDeadline(time.Time{})` перед возвратом. Любой новый код, который
временно ставит дедлайны на сокете, а затем отдаёт этот же сокет другому
потребителю (QUIC, будущий wireguard-go в Фазе 6) — обязан гарантировать
сброс дедлайна **после того как** все горутины, которые его выставляли,
точно завершились, а не просто по `defer` в месте, откуда дедлайн
изначально ставился.

### Фаза 6: mesh VPN, включая TUN/wireguard-go

Server-side координация (in-memory `NetworkRepository`, `usecase.JoinNetwork`
— авто-создание сети на фиксированном CIDR `100.64.0.0/10`, выдача
свободного IP, идемпотентный повторный join, `spur join-network` как
диагностика control-plane без данных) была готова первой.

Data-plane реализован через `internal/adapter/wgmesh`:
- `Bind`/`Endpoint` — своя реализация `conn.Bind` из `wireguard-go`,
  работающая НЕ через собственные UDP-сокеты wireguard-go, а поверх уже
  установленных `port.Stream`-каналов к каждому участнику сети (тех же,
  что даёт `usecase.EstablishSession` + `adapter/tunnel` для port-forward
  режима) — NAT traversal уже случился к моменту, когда у `Bind` появляется
  стрим, wireguard-go про punching/relay ничего не знает. Кадрирование
  пакетов поверх стрима — простой 2-байтный length-prefix (сами пакеты
  WireGuard уже аутентифицированы и зашифрованы его собственным Noise-
  протоколом, так что второй слой шифрования на этом стриме избыточен —
  и всё равно применяется через `adapter/e2e`, см. ниже, ради единообразия
  `rendezvous()`; накладные расходы двойного шифрования сочтены приемлемыми
  ради простоты, не обходить `rendezvous()` отдельной веткой).
- `Device` — создание TUN (`tun.CreateTUN`, требует root/`CAP_NET_ADMIN`),
  назначение адреса и подъём интерфейса через `ip addr add` / `ip link set
  up` (шелл-аут, не netlink-библиотека).
- `cmd/spur/mesh.go` — `join()`: присоединяется к сети на сервере, для
  каждого участника параллельно поднимает тоннель (переиспользуя
  `rendezvous()` из Фазы 5), берёт один поток на WireGuard-трафик,
  регистрирует в `Bind`, конфигурирует устройство через `IpcSet`
  (`BuildUAPIConfig`) с приватным ключом и списком peer'ов
  (public_key/allowed_ip=/32/endpoint=peer-id).
- Всё, что не требует root (`Bind`, кодирование UAPI-конфига,
  `domain.Network.NextAvailableIP`), покрыто юнит-тестами. Реальное
  создание TUN проверялось вручную с `sudo` через network namespaces на
  одном хосте (поскольку `spur join` жёстко использует имя интерфейса
  `spur0` — на разных реальных машинах это не конфликтует, а для
  теста на одном хосте каждый процесс был в своей `ip netns`, соединённых
  veth-парами). **Подтверждено живьём**: три узла (A, B, C, C
  присоединился позже A и B), настоящий TUN + WireGuard handshake +
  `ping` с 0% потерь во всех направлениях, включая C, подключившегося
  после того как A/B уже установили соединение друг с другом.

При живой проверке всплыло четыре реальных бага (все закрыты
регрессионными тестами, см. коммиты):
1. **Гонка при конкурентном join** (`usecase.JoinNetwork`): два пира,
   join'ящихся одновременно, читали сеть до того, как другой успевал
   сохранить свою запись — оба вычисляли один и тот же «следующий
   свободный IP», второй `Save` затирал первого. Фикс — атомарный
   `port.NetworkRepository.Update(name, mutate)` вместо раздельных
   `FindByName`+`Save`.
2. **Membership — не live-фид, а снимок на момент вызова**: если A
   join'ится раньше B, A просто не узнаёт о существовании B и никогда не
   пытается установить с ним тоннель — а B зависает навсегда, ожидая от A
   кандидатов, которые A никогда не опубликует. Фикс — `join()` теперь
   переопрашивает `JoinNetwork` каждые `meshRefreshInterval` (5с) и
   инкрементально подключается к новым участникам (`dev.IpcSet` без
   `replace_peers` — не трогает уже сконфигурированных пиров).
3. **`wireguard-go` вызывает `Close()`+`Open()` на `Bind` при каждом
   `listen_port=` в UAPI-конфиге** (`device/uapi.go`, `BindUpdate`
   безусловно ребиндит) — в том числе один раз при самой первой
   инициализации устройства, до того как добавлен хоть один пир. Первая
   версия `Bind.Close()` закрывала один и тот же канал `done` навсегда —
   после первого же ребинда приём пакетов ломался у всех пиров
   одновременно и бесшумно. Фикс — `done` пересоздаётся на каждый `Open()`
   отдельно от жизненного цикла пиров (стримы из `AddPeer` не зависят от
   Open/Close вообще).
4. **QUIC/yamux-стрим не появляется «на проводе», пока в него не
   запишут** — а WireGuard ничего не пишет, пока нет реального трафика
   (не проявляет инициативу просто потому, что peer сконфигурирован).
   Слушатель (`AcceptStream`) ждёт вечно стрим, который инициатор
   (`OpenStream`) никогда явно не «объявит». Фикс — `Bind.AddPeer`
   получает `isDialer`; инициатор сразу шлёт пустой кадр-«прайминг» сразу
   после открытия стрима, слушатель просто отбрасывает кадры нулевой
   длины как no-op.

Также выяснилось на этом же прогоне: дефолтный `HandshakeIdleTimeout`
quic-go (5с) слишком короткий при конкурентной нагрузке в этой
песочнице (несколько QUIC-соединений одновременно, планировщик под
нагрузкой) — увеличен до 30с в `infra.DefaultQUICConfig`.

  живьём: `--help` показывает подставленное из конфига значение по
  умолчанию, `connect` без `--server` уходит резолвить адрес из конфига,
  явный флаг всё равно перебивает файл.
- На момент закрытия дорожной карты (8 фаз) `SQLite`/`zerolog` из исходной
  таблицы стека ещё не были подключены — не требовались ни одной из фаз,
  задокументировано как сознательно отложенное, не баг. Оба закрыты сразу
  после по отдельному запросу — см. следующую секцию.

### Фаза 8+: персистентность сервера (SQLite)

По запросу пользователя после закрытия исходной дорожной карты — сервер
терял всё состояние (зарегистрированных пиров, mesh-сети и их участников)
на каждом рестарте, что нереалистично для узла, который должен работать
долго. `internal/adapter/repository/sqlite` реализует `port.PeerRepository`
и `port.NetworkRepository` поверх `modernc.org/sqlite` (чистый Go, без
cgo — то же требование к сборке, что и у остального модуля).

- `NetworkRepository.Update` держит тот же контракт атомарности, что и
  `memory.NetworkRepository` (см. «Фаза 6» выше): `sync.Mutex` вокруг всего
  read-mutate-write, а не просто SQL-транзакция — колбэк `mutate`
  (который вычисляет `domain.Network.NextAvailableIP` и может вернуть
  ошибку) выполняется в Go и не выражается одним SQL-выражением, так что
  транзакция сама по себе не даёт нужной атомарности без мьютекса поверх.
  Регрессионный тест (`TestNetworkRepository_ConcurrentUpdatesDontLoseMembers`)
  — тот же сценарий, что и `usecase.TestJoinNetwork_ConcurrentJoinsDontLoseMembers`,
  но прогнанный против настоящего SQLite-файла, а не карты в памяти.
- `CandidateStore` и `RelayBroker` остались в `adapter/repository/memory`
  осознанно, не по недосмотру: это по своей природе недолговечное,
  in-flight состояние одного обмена кандидатами или одной relay-сессии —
  персистить его между рестартами сервера бессмысленно, оно всё равно
  протухнет вместе с оборвавшимся QUIC-соединением клиента.
  `adapter/repository/memory` также остался лёгким тестовым дублёром
  `PeerRepository`/`NetworkRepository` в юнит-тестах usecase/adapter
  слоёв — настоящий SQLite-файл там был бы просто лишним трением.
- `spur server` получил флаг `--db` (по умолчанию —
  `infra.DefaultServerStatePath()`,
  `~/.config/spur/state.db`) — та же схема, что у `--identity`.
  **Живая проверка**: поднят сервер, зарегистрирован пир через
  `spur register`, сервер убит (`kill`, не graceful shutdown), содержимое
  строки подтверждено напрямую через `sqlite3 state.db "select ..."` —
  строка на диске. Отдельно — `spur join-network` от двух разных identity
  (A создаёт сеть и получает инвайт-токен, B входит по токену), сервер
  убит и поднят заново на том же `--db`, A повторно вызывает
  `join-network` без токена (т.к. уже участник) — видит себя и B в
  membership без потери данных.

### Фаза 8+: структурные логи (zerolog)

По тому же запросу — до этого каждый обработчик control-plane
(`handleRegister`, `handlePublishCandidates`, `handleRelay`,
`handleJoinNetwork` и т.д.) тихо дропал любую ошибку: битый фрейм от
клиента, ошибку usecase, невалидный инвайт-токен — снаружи ничего не
было видно, оператор долгоживущего сервера не мог отличить «клиент
отвалился штатно» от «что-то реально сломано» без `strace`/`tcpdump`.

- `controlserver.Server.Logger *zerolog.Logger` — не встроенное поле по
  значению: zero-value `zerolog.Logger{}` пишет в nil `io.Writer` и
  паникует при первом реальном логе, а десятки существующих тестов
  собирают `&controlserver.Server{...}` без `Logger` вообще. Указатель +
  `log()`-хелпер (`nil` → пакетный `nopLogger = zerolog.Nop()`) держит
  все старые тесты рабочими без единой правки — логирование просто
  молча выключено, если вызывающий его не подключил.
- Уровни: `Info` — заметные для оператора события (пир зарегистрировался,
  пир вошёл в mesh-сеть, relay-сессия открылась — punching не удался,
  трафик пошёл через сервер, это операционно значимо, не шум); `Warn` —
  подозрительные, но не фатальные вещи (битый фрейм, невалидный
  инвайт-токен — это ещё и потенциальный сигнал перебора токенов, стоит
  видеть по умолчанию, не только в debug); `Error` — сам usecase упал;
  `Debug` (только с `--verbose`) — рутинный трафик по каждому RPC
  (`accept`, publish/await candidates), который на info-уровне была бы
  просто шумом на каждое обычное подключение.
- `spur server --verbose` переключает уровень с info на debug
  (`infra.NewLogger(debug bool)`); без флага — только info и выше.
  **Живая проверка**: без `--verbose` в выводе только `INF`-строки
  (peer registered/joined), с `--verbose` появляются `DBG`-строки
  (connection accepted); отдельно спровоцирован реальный неверный
  инвайт-токен против уже существующей (не только что созданной) сети —
  в логе `WRN join-network: invalid invite token`.

### Разделение клиента и сервера на два бинарника

По запросу пользователя: раньше `spur server` был подкомандой одного
бинарника, и клиентские команды (`connect`, `join`, ...) неизбежно
линковали в себя весь серверный стек — SQLite-драйвер
(`modernc.org/sqlite`), `controlserver`, `stunserver`,
`adapter/repository/{memory,sqlite}` — хотя ни разу их не вызывали.
Go-линкер не может выкинуть эти пакеты как мёртвый код: они достижимы из
`main()`, поскольку одна и та же `cli.NewRootCommand` собирала все
подкоманды разом, включая `server`.

- `internal/adapter/cli` теперь строит два разных дерева команд:
  `NewClientRootCommand(ClientDependencies, ClientDefaults)` (`whoami`,
  `connect`, `expose`, `join`, `join-network`, `register`, `version`) и
  `NewServerRootCommand(ServerDependencies,
  ServerDefaults)` (сервер прямо на корневой команде — раз в бинарнике
  больше нечего делать, подкоманда `server` не нужна — плюс `version`).
  Старые единые `Dependencies`/`Defaults` разъехались по смыслу
  использования, не по слою: это по-прежнему один и тот же адаптер
  (`internal/adapter/cli`), просто с двумя точками входа, так же как
  `adapter/tunnel` даёт один транспорт на оба режима данных.
- `cmd/spur/main.go` — теперь только клиентский composition root:
  `register`, `whoami`, `connect`, `expose`, `join`, `joinNetwork`. Не
  импортирует `controlserver`, `stunserver`, `adapter/repository/*`
  вообще.
- `cmd/spur-server/main.go` — новый, отдельный composition root: `runServer`
  переехал сюда как есть (SQLite-репозитории, in-memory
  candidate/relay-брокеры, `controlserver.Server`, `stunserver.Serve`).
- Оба `cmd/*` — независимые `main`-пакеты, ни один не импортирует
  другой; `internal/*` остаётся общим для обоих, как и раньше.
- `Makefile`: `build`/`release` теперь собирают оба бинарника
  (`bin/spur`, `bin/spur-server`; `dist/spur-<os>-<arch>`,
  `dist/spur-server-<os>-<arch>`).
- **Проверено живьём**: `go list -deps ./cmd/spur` больше не содержит
  `modernc.org/sqlite`/`controlserver`/`stunserver`/`adapter/repository/*`
  ни в каком виде; `go list -deps ./cmd/spur-server` не содержит
  `wireguard-go` (он нужен только клиентскому `join`). Размер
  клиентского бинарника упал с ~16.7 МБ (когда `spur server` был частью
  того же бинарника) до ~12.5 МБ. `spur-server` поднят отдельно,
  `spur whoami`/`spur register` против него отработали штатно; `spur
  --help` подтверждает отсутствие `server` среди подкоманд, `spur-server
  --help` показывает флаги (`--listen`, `--stun-listen`, `--db`,
  `--verbose`) прямо на корневой команде.

### Режим передачи файлов и директорий (`spur send`/`spur receive`)

По тому же запросу пользователя, что и разделение бинарников —
третий data-plane режим поверх уже существующего rendezvous/punching
слоя (та же `rendezvous()`, что у `connect`/`expose`/`join`), но в
отличие от них не долгоживущий: одна отправка = одна операция с чётким
концом, а не постоянно открытый канал.

- `internal/usecase/port/filesystem.go` — новые порты `FileSource`
  (`List`/`Open`, сторона `send`) и `FileSink` (`Create`, сторона
  `receive`), по образцу `LocalListener`/`LocalDialer` у port-forward.
- `internal/usecase/{send_files,receive_files,file_transfer_wire}.go` —
  `SendFiles`/`ReceiveFiles` открывают/принимают ровно один стрим на
  весь трансфер (не цикл на много подключений, как `ForwardPort`), пишут
  простое кастомное кадрирование без внешних зависимостей: `pathLen
  (uint16) path size(uint64) content`, завершается заголовком с
  `pathLen == 0`. Протестировано на моках (`FileSource`/`FileSink` в
  памяти + `net.Pipe()` вместо реального туннеля) до того, как появился
  adapter — по правилу «сначала порт и usecase с тестами на моках».
- `internal/adapter/localfs` — реализация портов поверх настоящей ФС.
  `Sink.Create` **обязательно** санитизирует `RelPath` (`filepath.Clean`
  + проверка на `..`/абсолютный путь) перед `filepath.Join` с целевой
  директорией: `RelPath` приходит по сети от пира, а не из доверенного
  локального источника — без проверки вредоносный или сбойный
  отправитель мог бы писать вовне `--out` (`../../.ssh/authorized_keys`
  и т.п.). Регрессионный тест `TestSink_RejectsPathTraversal`.
- `spur send <path> --to <peer>` / `spur receive --to <peer> --out <dir>`
  в `cmd/spur/filetransfer.go`, переиспользуют `rendezvous()` как есть.

**Реальный баг, найденный живой проверкой** (не ловится юнит-тестами —
там `net.Pipe()` полностью синхронный, гонка физически не может
произойти): при передаче большого файла (500КБ+, но не совсем
маленького) соединение обрывалось с ошибкой QUIC `Application error
0x0 (remote)`, при этом маленький файл, отправленный первым, доходил
целиком. Причина — `SendFiles.Run` изначально возвращался сразу после
последней записи в стрим, `send()`-обвязка тут же вызывала
`tun.Close()` (закрывает всё QUIC/yamux-соединение целиком, не только
стрим). `Stream.Close()` в quic-go шлёт FIN и запускает flush
асинхронно, сам не блокируется до фактической доставки — закрытие
родительского соединения сразу вслед за этим может оборвать ещё не
доставленные данные. Все остальные data-plane режимы этого не ловили,
потому что держат туннель открытым на весь срок жизни процесса и
никогда не закрывают его «сразу после одной операции» — `send`/`receive`
первые, кто это делает.

Фикс — явный ack на уровне протокола, симметричный с обеих сторон:
после `pathLen == 0` `ReceiveFiles` пишет один байт-подтверждение и
только затем блокируется на чтении (что угодно — чистый EOF или ошибка
соединения — сигнализирует «отправитель закрылся, теперь можно
закрываться и мне») перед `return`; `SendFiles` не возвращается, пока
не прочитает этот байт. Без второй половины (receiver ждёт до своего
close) баг воспроизводился заново уже на пути подтверждения — тот же
эффект, просто на один байт вместо файла, и именно живая проверка (не
юнит-тесты на `net.Pipe()`) снова была единственным способом это
поймать. Проверено вручную: файл 2МБ + маленькие файлы в одной
передаче, оба процесса (`send`/`receive`) выходят с кодом 0, `diff -r`
между исходной и принятой директорией — `IDENTICAL`.

### `install.sh`: автоматическая установка в PATH

Первая попытка закрыть этот запрос была через переменные окружения
(`SPUR_SERVER` и т.п.) — неверно поняли задачу, откачено полностью
(коммит `aacc4d4`, см. сообщение коммита), не оставляя мёртвого кода.
Реальный запрос — чтобы `spur`/`spur-server` становились доступны как
обычные команды на любой машине без ручной правки rc-файлов.

Важное ограничение, которое нельзя обойти в принципе: запущенный
процесс не может изменить `PATH` уже открытого родительского шелла —
переменные окружения не текут «вверх» по дереву процессов. Поэтому
«автоматически» здесь значит: скрипт сам определяет шелл пользователя
и дописывает нужную строку в его rc-файл — эффект появляется в новых
сессиях шелла (или после `source`), не в той, что уже запустила
установку. Это же ограничение есть у rustup/nvm/homebrew и любого
похожего инсталлятора — не особенность этого проекта.

`install.sh` (корень репозитория, POSIX `sh`, `set -eu`):
- Определяет ОС/архитектуру через `uname`, скачивает `spur-<os>-<arch>`
  и `spur-server-<os>-<arch>` с `releases/latest/download/...` (не
  требует Go — рассчитан именно на «голую» машину). Если задан
  `SPUR_LOCAL_BUILD_DIR` (так его вызывает `make install`) — копирует
  уже собранные локально бинарники вместо скачивания, чтобы не
  дублировать логику настройки PATH между двумя сценариями.
- Ставит в `$SPUR_INSTALL_DIR` (по умолчанию `~/.local/bin`).
- Если каталог уже есть в `$PATH` — ничего в rc-файлы не пишет и молча
  завершается (типичный случай на многих дистрибутивах, где
  `~/.local/bin` уже в `PATH` по умолчанию).
- Иначе определяет шелл через `$SHELL` (`bash`/`zsh`/`fish`) и
  добавляет строку в соответствующий rc-файл (`~/.bashrc` или
  `~/.bash_profile`, если он уже существует — macOS-конвенция для
  login-шеллов; `~/.zshrc`; `~/.config/fish/config.fish` через
  `fish_add_path`), идемпотентно — проверка `grep -qF` перед записью,
  повторный запуск не плодит дубликаты.
- Windows не поддержан (POSIX `sh` там в принципе не при делах) —
  печатает инструкцию скачать `.exe` и добавить папку в `PATH` вручную
  через «Переменные среды».
- `Makefile`: `install` = `build` + `SPUR_LOCAL_BUILD_DIR=$(BIN_DIR)
  ./install.sh` — один и тот же скрипт для «curl | sh» на чужой машине
  и для локальной сборки из исходников, а не две отдельные реализации
  одного и того же PATH-хука.

**Проверено живьём**: сценарии bash/zsh/fish в изолированном `$HOME`
(временная директория, чтобы не трогать реальные dotfiles при
тестировании) — каждый шелл получил правильную строку в свой rc-файл,
повторный запуск не задублировал; отдельно проверен случай «каталог уже
в PATH» (ничего не пишется) и реальная загрузка бинарников с
`github.com/fu1se/SPUR/releases/latest` (не заглушка). После этого
`make install` прогнан по-настоящему на этой машине: убраны прежние
артефакты предыдущего (ошибочного, env-based) подхода — `fish_user_paths`
c `~/go/bin` и старые бинарники там же — и `spur version` /
`spur-server --help` отработали как обычные команды из свежей
fish-сессии, ставя `~/.local/bin` (который на этой системе и так уже
был в `PATH` — типичный случай, описанный выше).

### Аудит безопасности/надёжности по запросу пользователя (пост-Фаза 8+)

По запросу пользователя проведён полный аудит проекта (параллельные
агенты по подсистемам: control-plane/crypto, NAT/tunnel/relay, mesh VPN/
persistence, CLI/infra/file-transfer) и затем последовательно исправлено
всё найденное — 10 коммитов, каждый со своими регрессионными тестами.
Самое важное:

- **Переиспользование ключа/nonce в `adapter/e2e`** (критично): ключи
  шифрования выводились только из статичных identity-ключей без соли —
  один и тот же ключ на каждую сессию между теми же двумя пирами
  навсегда, а nonce-счётчик каждого `Stream` стартовал с 0 — два
  параллельных стрима на одном соединении шифровали первый кадр
  одинаковым (key, nonce). Первая попытка фикса (общий atomic-счётчик на
  все стримы одного `wrappedConn`) оказалась сама с багом, пойманным
  собственным regression-тестом под `-race` через несколько прогонов:
  `usecase.pipe` гоняет каждый стрим в своей горутине, и порядок, в
  котором разные стримы разбирают общий счётчик на приёме, не гарантированно
  совпадает с порядком, в котором отправитель их зашифровал — расшифровка
  падала с `cipher: message authentication failed`. Реальный фикс —
  свой nonce ID на стрим (счётчик уровня `wrappedConn`, не общий на
  подсчёт, а просто гарантирующий уникальность), передаётся один раз как
  4-байтный префикс вместе с первым `Write()` (не сразу при создании
  стрима — иначе блокировка на синхронных транспортах, пока пир ещё не
  читает), плюс отдельный per-stream счётчик кадров. Nonce = `[4 байта
  ID][8 байт счётчика]` — уникален и между стримами (за счёт ID), и
  внутри одного стрима (счётчик трогает только сам этот стрим). Урок:
  **любой «общий примитив на несколько независимых читателей/писателей»
  (общий счётчик, общий канал) может незаметно завязать корректность на
  порядок, которого система не гарантирует — проверять `-race` с
  несколькими прогонами, не одним**.
- **Подмена личности при `PublishCandidates`**: сервер доверял
  клиент-заявленному `peer_id` вместо вывода его из присланного публичного
  ключа (как уже делает `handleJoinNetwork`) — зная чужой `peer_id`
  (публичный, не секрет), можно было опубликовать поддельные кандидаты
  под чужим именем и перехватить сессию. Исправлено: `peer_id` теперь
  всегда выводится из `public_key` на сервере.
- **Спуфинг NAT-traversal**: `UDPPuncher.recvLoop` принимал punch-ответ
  от любого адреса, не только из списка предложенных кандидатов;
  `DiscoverServerReflexive` не проверял ни источник STUN-ответа, ни
  `TransactionID`. Оба исправлены с проверкой источника/транзакции.
- **Утечки памяти в брокерах** (`CandidateBroker`, `RelayBroker`):
  ни один не чистил свою запись при абандоне/таймауте — любой
  неаутентифицированный клиент мог растить карты сервера без ограничения
  случайными `session_id`. Добавлены немедленная очистка при
  потреблении/отмене плюс TTL-развёртка для orphaned `Put`. `RelayBroker`
  вдобавок вообще не имел таймаута на ожидание пары — при асимметричном
  исходе punching (реальный сценарий, не только атака) сторона, ушедшая
  на relay, могла зависнуть навсегда без единой ошибки.
- **wgmesh**: пять связанных проблем жизненного цикла пира — успешный
  `IpcSet` больше не требуется задним числом (peer помечается
  подключённым только после реального успеха), `Bind` умеет забывать
  скончавшийся стрим сам (`readLoop`'s self-cleanup) и `cmd/spur`
  переподключается на следующем тике вместо вечного пропуска,
  повторный `AddPeer` закрывает старый стрим вместо утечки горутины,
  общий канал приёма пакетов на всех пиров заменён на канал на пира (один
  шумный пир больше не блокирует доставку остальным через `reflect.Select`
  fan-in), мёртвая проверка размера пакета (`> 65535` для `uint16`) заменена
  на реальный предел.
- **`localfs` (send/receive)**: симлинки в отправляемом дереве больше не
  разыменовываются (утечка контента произвольного файла под чужим именем
  — подтверждено живым воспроизведением), приёмник теперь проверяет, что
  реальный (через `EvalSymlinks`) путь записи остаётся внутри `DestDir`
  даже если там уже есть заранее подложенный симлинк.
- Более мелкое: `WriteFrame` теперь проверяет `maxFrameSize` (не только
  `ReadFrame`), invite-токен сравнивается constant-time, SQLite больше не
  перезаписывает таблицу участников на no-op heartbeat re-join, добавлены
  разумные пределы конкурентности на `controlserver` (стримы) и
  `ForwardPort`/`ServeExposedPort` (локальные соединения).

Полные обоснования, сценарии эксплуатации и regression-тесты — в
сообщениях соответствующих коммитов (`git log`, начиная с
`ef86b11 fix: harden e2e session keys...` по
`1f63e1c fix: bound concurrency...`).

### Прогресс-бар и скорость для `send`/`receive`

По запросу пользователя — раньше `send`/`receive` молчали до самого
конца передачи, никакой обратной связи в процессе.

- `internal/usecase/progress.go` — `TransferProgress` (callback-тип,
  nil-safe, тот же паттерн, что `controlserver.Server.Logger`) и
  `copyWithProgress` — замена `io.CopyN` на копирование чанками по
  32 КБ с вызовом колбэка после каждого чанка. `SendFiles`/`ReceiveFiles`
  получили поле `OnProgress TransferProgress`.
- Отправитель знает полный манифест (`Source.List()`) заранее — может
  показать честный `overallTotal` и процент по всей передаче. Получатель
  узнаёт файлы по одному из заголовков на лету (см. `file_transfer_wire.go`)
  и не знает суммарный размер заранее — сознательно не стал менять
  wire-протокол ради этого (добавлять "манифест" заголовком в начало),
  особенно после недавнего болезненного опыта с рассинхроном клиент/сервер
  на `salt`-поле — лишний повод сломать совместимость того не стоил.
  Получатель просто показывает бегущий счётчик байт вместо процента.
- Рендеринг самой строки прогресса — **в `internal/adapter/cli`**
  (`progress.go`: `newProgressPrinter`, `humanBytes`), не в `cmd/spur`:
  тот же принцип, что уже применялся к `onSelfID` — `cmd/spur`
  (composition root) не занимается терминальным выводом сам, а
  прокидывает colback дальше; весь `cmd.Printf`/форматирование живёт в
  `adapter/cli`. Ради этого `cli.ClientDependencies.Send`/`.Receive`
  получили новый параметр `onProgress cli.ProgressFunc` — свой тип в
  `cli`, не алиас на `usecase.TransferProgress` (тот же принцип, что у
  `RegisterResult`/`JoinNetworkResult` — `cli` не должен зависеть от
  `usecase`).
- Троттлинг перерисовки — раз в 100мс (кроме последнего чанка файла,
  который показывается всегда), скорость — скользящее окно 500мс, не
  мгновенная (иначе слишком дёргано на быстрых локальных передачах).
  Строка перерисовывается на месте (`\r` + `\033[K` для очистки хвоста
  при укорачивании) — тот же ANSI-подход, что уже использует
  `zerolog.ConsoleWriter` для логов сервера, ничего нового для терминала
  не требуется.
- **Проверено живьём**: реальная передача файла 30 МБ через `script`
  (псевдо-tty, чтобы `\r`/ANSI реально сработали, не потерялись в
  редиректе) — на обеих сторонах видна обновляющаяся строка с процентом,
  скоростью и итогом, `diff` между исходным и принятым файлом —
  идентичны, оба процесса завершились с кодом `0`.

### `yamux`-таймаут keepalive рвал большие relay-передачи

Найдено пользователем на реальной передаче: `spur send` директории на
~101 ГБ упал на 19 ГБ (relay-путь, punching не прошёл) с
`yamux: keepalive failed: i/o deadline reached` и
`Error: usecase: send ...: stream closed` — притом что соединение до
этого стабильно держало ~7 МБ/с несколько минут, то есть было живым.

Причина — в самой библиотеке `hashicorp/yamux`: `adapter/tunnel`
вызывал `yamux.Client`/`yamux.Server` с `nil`-конфигом, то есть с
`yamux.DefaultConfig()`, где `ConnectionWriteTimeout` — 10 секунд.
Keepalive-пинг `yamux` кладёт в ту же очередь отправки, что и обычные
данные стрима (`Session.keepalive` → `Ping` → `waitForSendErr`, общий
`sendCh`) — на relay-сессии, которая параллельно гонит гигабайты
файлового трансфера, пинг вполне может не протолкнуться через занятую
очередь за 10 секунд, даже если соединение физически живо. `yamux`
трактует это как мёртвое соединение и рвёт **всю** сессию немедленно
(`Session.keepalive` → `exitErr(ErrKeepAliveTimeout)`) — никакого
плавного возобновления, вся передача обрывается.

Фикс — `internal/adapter/tunnel/transport.go`'s `yamuxConfig()`:
свой `*yamux.Config` вместо `nil`, `ConnectionWriteTimeout` увеличен до
60 секунд (даёт запас на реальную, но временную перегрузку стрима
большим файлом, не трогая обнаружение по-настоящему мёртвых соединений
— просто менее нервно). Заодно `LogOutput: io.Discard` — `yamux` по
умолчанию пишет через stdlib `log` прямо в stderr необорудованным
текстом (`2026/07/03 15:20:54 [ERR] yamux: ...`), не в стиле остального
вывода проекта; ошибка и так корректно всплывает через возвращаемый
`error` у соответствующего `Read`/`Write`, дублирующий лог только шумит.
Регрессионный тест (`TestYamuxConfig_ToleratesSlowWrites`, white-box —
`yamuxConfig()` неэкспортирован) проверяет сам факт увеличенного
таймаута и отключённого лога; полноценно смоделировать 10+ секундную
перегрузку сети в юнит-тесте без раздувания времени прогона всего
набора признано непропорциональным — сама логика воспроизведена и
проверена по исходникам `yamux`, а не вслепую.

### Дружелюбные сообщения об ошибках (`cli.Explain`)

По запросу пользователя — раньше любая ошибка долетала до терминала как
сырая цепочка `fmt.Errorf("app: <шаг>: %w", err)`, например `Error: app:
exchange candidates: controlproto: read header: EOF`. Понятно только тому,
кто читал код.

- `internal/adapter/cli/friendlyerror.go` — `Explain(err) string`
  сопоставляет известные «шаги» (они уже стабильно помечены префиксом
  `"app: <шаг>: ..."`/`"server: <шаг>: ..."` в `cmd/spur/tunnel.go`,
  `cmd/spur/mesh.go` и т.п. — сопоставление по строке, а не заведение
  типизированных sentinel-ошибок на каждый call site, потому что префиксы
  свои, контролируемые, и есть безопасный fallback) плюс реальную причину
  там, где это важно (`errors.Is(err, context.DeadlineExceeded)`,
  `errors.Is(err, fs.ErrNotExist)`, `io.EOF`) — и выдаёт заголовок
  человеческим языком + короткую подсказку что проверить + **всегда**
  исходный технический текст (`Технические детали: ...`), ничего не
  прячется.
- Оба root-command (`NewClientRootCommand`/`NewServerRootCommand`)
  переключены на `SilenceErrors: true` — печать ошибки теперь не отдана
  cobra, а делается явно в `cmd/spur/main.go`/`cmd/spur-server/main.go`
  после `root.ExecuteContext`, через `cli.Explain(err)`. Это позволило
  подключить перевод в одном месте на бинарник, не трогая все `RunE` по
  отдельности.
- Покрыто конкретными сценариями, реально встреченными в этой сессии на
  практике (не выдуманными): STUN-таймаут, EOF при обмене кандидатами
  (не тот `--to`/собеседник не успел), обрыв relay/yamux keepalive,
  `address already in use` (частая причина — одинаковые `--listen` и
  `--stun-listen`), неверный инвайт-токен, файл/директория не найдены
  (с подсказкой конкретно про `./mnt/...` vs `/mnt/...` — реальная
  ошибка пользователя из этой же сессии).

### Ctrl+C с подтверждением

По запросу пользователя — раньше единственный `Ctrl+C` сразу рвал
`context.Context` через `signal.NotifyContext`, что при долгой передаче
файла/mesh-сессии больно от случайного нажатия.

- `internal/infra/interrupt.go` — `ContextWithConfirmedInterrupt(parent,
  window, warn)`: первый `SIGINT` не отменяет контекст, только вызывает
  `warn` (nil-safe) и запускает окно (`3с` в обоих `cmd/*`); второй
  `SIGINT` в пределах окна — реальная отмена; если окно истекло без
  второго нажатия — состояние сбрасывается, снова нужны два нажатия «с
  нуля». `SIGTERM` отменяет сразу на первом сигнале без подтверждения —
  он приходит от process manager/`kill`, а не от случайной клавиши,
  подтверждать нечего.
- Используется в обоих `cmd/spur/main.go` и `cmd/spur-server/main.go`
  вместо `signal.NotifyContext` — общий хелпер, а не дублирование логики
  на два бинарника.
- Тесты (`internal/infra/interrupt_test.go`) шлют настоящие OS-сигналы
  самому тестовому процессу (`syscall.Kill(os.Getpid(), syscall.SIGINT)`)
  — не мокают сигналы, проверяют реальное поведение `signal.Notify`:
  одиночное нажатие не отменяет, два в окне — отменяют, два вне окна —
  не отменяют (окно истекло), `SIGTERM` — отменяет сразу, `warn`
  вызывается ровно один раз на первое нажатие. **Проверено живьём**:
  реальный `spur-server` процесс, `kill -INT` один раз — жив, `kill
  -INT` второй раз в течение 3 секунд — корректно завершается.

### Короткие коды подключения + одна команда вместо `whoami`+`connect`

По запросу пользователя («упрости подключение») — раньше обе стороны
были обязаны сначала независимо запустить `whoami`, обменяться
32-символьными hex peer-id вручную (в этой же сессии несколько раз
приводило к опечаткам/не тем `--to`), и только потом запускать
`connect`/`expose`/`send`/`receive` с верным ID с обеих сторон.

Осознанно **не** стал укорачивать сам peer-id (усечённый хэш был бы
эксплуатируем — коллизия в 24-32 битах вместо 128 подбирается перебором,
и тогда чужой короткий код мог бы совпасть с кодом жертвы, подставив
атакующего вместо реального адресата). Вместо этого — отдельный,
недолговечный, посреднический механизм на сервере:

- **Протокол** (`proto/control/v1/control.proto`): три новых RPC —
  `RegisterPairingCode` (клиент → сервер: «вот мой публичный ключ, дай
  мне код»), `ResolvePairingCode` (клиент → сервер: «вот код и мой
  публичный ключ, кто за ним стоит» — заодно регистрирует вызывающего
  как «использовавшего» код), `AwaitPairingCodeUse` (клиент → сервер:
  блокируется, пока кто-нибудь не сделает `Resolve` для этого кода,
  возвращает peer-id того, кто это сделал). Как и у `PublishCandidates`/
  `JoinNetwork`, peer-id везде выводится сервером из публичного ключа,
  не доверяется клиент-заявленному значению — та же защита от подмены
  личности.
- **`internal/usecase/pairing.go`**: `RegisterPairingCode` генерирует
  6-символьный код через `crypto/rand` из алфавита Crockford base32 без
  визуально похожих символов (`0/O`, `1/I/L`, без `U`) — читается и
  печатается человеком без ошибок транскрипции, в отличие от peer-id,
  который никогда не предназначался для устного/визуального обмена.
  32 символа алфавита ровно делят 256 без остатка — `byte % 32` даёт
  честное равномерное распределение без необходимости в отбраковке
  (rejection sampling). TTL — `PairingCodeTTL = 10 минут` (экспортирован,
  используется и клиентом, и сервером как таймаут `AwaitPairingCodeUse`).
- **`internal/adapter/repository/memory/pairing_code_broker.go`** —
  in-memory, тот же паттерн оппортунистической TTL-очистки, что у
  `CandidateBroker` (без фонового таймера — очистка на каждый вызов).
  `Resolve`/`AwaitUse` соединены через `chan domain.PeerID` ёмкостью 1
  на код — первый `Resolve` будит ожидающий `AwaitUse`, повторные
  `Resolve` того же кода не блокируются (не только первое использование
  кода валидно — код может использоваться для диагностики/повторной
  попытки, но будит `AwaitUse` только один раз, что соответствует
  ожидаемому сценарию «один код — одно подключение»).
- **Клиентский флоу** (`cmd/spur/tunnel.go`) — `rendezvous()` стал
  принимать не сам `domain.PeerID`, а `counterpartResolver`
  (`func(ctx, client, id) (domain.PeerID, error)`) — единая точка, через
  которую проходят все четыре варианта того, «как узнать, с кем
  разговариваем»: `resolveCounterpartArg` (гость: строка похожа на
  peer-id по форме — 32 строчных hex-символа — использует как есть, иначе
  резолвит как код через сервер; `looksLikePeerID` отличает форматы без
  похода на сервер), `hostViaPairingCode` (хост: регистрирует код,
  печатает его через колбэк, блокируется на `AwaitPairingCodeUse`),
  `fixedCounterpart` (mesh-режим: counterpart уже известен из membership
  сети, резолвить нечего). Всё, что происходит после резолва
  counterpart'а (обмен кандидатами, punching/relay, e2e-шифрование) —
  ни строчки не изменилось, что и было целью рефакторинга через общий
  интерфейс вместо дублирования `rendezvous()` на хост/гостя.
- **CLI**: `--to` стал необязательным у `connect`/`expose`/`send`/
  `receive` — пустое значение включает «режим хоста». Печать кода —
  через новый `cli.OnCodeFunc`-колбэк (`internal/adapter/cli/pairing.go`,
  `newCodePrinter`), по тому же принципу, что уже применялся к
  `onSelfID`/`ProgressFunc`: `cmd/spur` (composition root) не занимается
  терминальным выводом сам, только прокидывает колбэки, всё форматирование
  — в `adapter/cli`.
- **Проверено живьём**: `spur send` без `--to` печатает код, `spur
  receive --to <код>` резолвит его и подключается — `session_id`
  совпадает на обеих сторонах в логах сервера, файл дошёл побайтово
  идентично. Отдельно — `connect`/`expose` с кодом на реальном
  TCP-эхо-сервисе (round trip корректен). Отдельно — старый флоу с
  полным peer-id (`whoami` на обеих сторонах + явный `--to`) по-прежнему
  работает без изменений — полная обратная совместимость.

### Докачка прерванной передачи файлов

По тому же четырёхчастному запросу пользователя, что и friendly-ошибки/
Ctrl+C-подтверждение (см. секции выше) — последняя, четвёртая часть:
если `spur receive` прервался на середине (крах, `Ctrl+C`, обрыв связи),
повторный запуск начинал всё с нуля, даже если 99% файла уже лежало на
диске.

Это **breaking change wire-протокола** (как и добавление `salt` —
`ef86b11`, см. выше): обе стороны должны обновиться вместе.

- **Протокол** (`internal/usecase/file_transfer_wire.go`) стал
  трёхфазным вместо однопроходного:
  1. **Манифест** — отправитель шлёт заголовки (`pathLen`+`path`+`size`)
     всех файлов подряд, завершая маркером `pathLen == 0` — ровно как
     раньше, но теперь единым блоком в начале, а не по одному файлу
     вперемешку с содержимым.
  2. **План докачки** — получатель, зная полный список файлов и
     заранее проверив `port.FileSink.ExistingSize` для каждого,
     отправляет по одному `uint64`-офсету на файл, в том же порядке,
     что и в манифесте (0 — качать с нуля, `entry.Size` — уже всё есть).
  3. **Содержимое** — отправитель шлёт остаток каждого файла, начиная с
     полученного офсета (`port.FileSource.Open(relPath, skip)`), без
     дополнительного покадрового заголовка (позиция в потоке уже
     однозначно определена манифестом+планом с обеих сторон).
- **`internal/usecase/port/filesystem.go`**: `FileSource.Open` получил
  второй параметр `skip int64` (переоткрыть с нужной позиции, а не
  всегда с начала); `FileSink` получил `ExistingSize(relPath) (int64,
  error)` (0 для отсутствующего файла) и `Create` — второй параметр
  `offset int64` (0 — как раньше, truncate; >0 — дозапись строго с этой
  позиции).
- **`internal/adapter/localfs`**: `Source.Open` делает `Seek` при
  `skip > 0`; `Sink.ExistingSize` — `os.Stat`, `os.IsNotExist` → `(0,
  nil)`; `Sink.Create` при `offset > 0` **перепроверяет**, что реальный
  размер файла на диске совпадает с заявленным офсетом, прежде чем
  открыть на дозапись (`os.O_WRONLY`+`Seek`) — не доверяет офсету
  вслепую: рассинхрон (например, файл кто-то тронул между
  `ExistingSize` и `Create`) иначе означал бы либо дыру в файле, либо
  затирание байтов, которые должны были остаться.
- **`usecase.ReceiveFiles.OnResumeOffer`** (тип `ResumeOffer func(filesWithData
  int, alreadyHave, total int64) bool`) — спрашивается **один раз на всю
  передачу**, не по каждому файлу отдельно: типичный сценарий — одна
  прерванная передача, а не россыпь несвязанных одноимённых файлов по
  отдельности. `nil` (как и явное `false`) — прежнее поведение, всегда
  с нуля. Файлы нулевого размера — отдельный граничный случай: у них
  `resumeFrom == entry.Size` тривиально верно (`0 == 0`) даже при
  чистом старте, что по невнимательности сначала приводило к тому, что
  `Sink.Create` для них вообще не вызывался и пустой файл не создавался
  на диске — поймано юнит-тестом
  (`TestSendReceiveFiles_MultipleFilesPreserveRelativePaths`, которая уже
  включала файл нулевого размера) сразу после рефакторинга, до всякой
  живой проверки. Фикс — пропускать фазу передачи содержимого только при
  `entry.Size > 0 && resumeFrom[i] == entry.Size`.
- **CLI**: `internal/adapter/cli/resume.go` — `newResumePrompt` печатает
  вопрос в stderr (`cmd.ErrOrStderr()`) и читает ответ из
  `cmd.InOrStdin()`; пустой ввод (просто Enter) или `y`/`yes`/`д`/`да`
  (без учёта регистра) — «да», всё остальное, включая ошибку чтения
  (например stdin не терминал) — «нет», а не бесконечное ожидание ответа,
  который никогда не придёт. `cli.ResumeOfferFunc` — зеркальный тип
  `usecase.ResumeOffer` (та же причина, что у `ProgressFunc`/`OnCodeFunc`
  — `cli` не имеет права импортировать `usecase` напрямую), подключён
  только у `spur receive` (`internal/adapter/cli/receive.go`) — у
  `send` спрашивать нечего, план докачки решает получатель.
- **Проверено живьём**: полная передача 8 МБ файла — побайтово идентична
  исходнику. Затем `truncate` файла на приёмнике до 3 МБ (имитация
  краха на середине) + повторный `send`/`receive` с ответом «да» на
  запрос — прогресс-бар получателя стартует сразу с 38% (не с 0%),
  итоговый файл снова побайтово идентичен исходнику. Отдельно — та же
  имитация, но с **испорченным** (не просто усечённым, а перезаписанным
  нулями) сохранённым префиксом и ответом «нет» — получатель стартует с
  0%, итоговый файл всё равно побайтово идентичен исходнику: отказ от
  докачки не доверяет старому содержимому, полностью его перезаписывает
  поверх, а не просто помечает как «начать сначала» без реальной
  перезаписи.
